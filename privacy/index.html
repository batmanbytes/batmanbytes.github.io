The Transparency Problem

Bitcoin's UTXO model stores every output, every amount, every spending relationship — forever, publicly.

The three core heuristics

  • Common Input Ownership (CIOH) — multiple inputs presumed from the same wallet. Powers most address clustering.
  • Change detection — script type, round amounts, output ordering reveal which output is the recipient vs. the change.
  • Address reuse — trivially links all payments to the same identity.

The industry that exploits it

  • Chainalysis, Elliptic, Scorechain, TRM Labs
  • Cluster addresses → link to real identities via KYC leakage
  • Score UTXOs as "tainted" — exchanges act on these scores
  • Direct government contracts: DOJ, IRS-CI, OFAC

CoinJoin: the Core Idea

Multiple users combine inputs into a single transaction, making CIOH ambiguous.

Without CoinJoin:

Alice: UTXO_A → Payment_to_Bob + Change_A

With CoinJoin (n participants):

UTXO_A + UTXO_B + ... + UTXO_N → Out_1 + Out_2 + ... + Out_N
  • No consensus changes needed — works on Bitcoin today
  • Key challenge: coordination without a trusted party, and without leaking who-gets-what

WabiSabi

Keyed-verification anonymous credentials for variable-amount CoinJoin

WabiSabi: From ZeroLink to KVAC

ZeroLink / Wasabi v1 (2018)

  • Chaumian blind signatures
  • Coordinator issues blind sig over an output credential
  • Registration unlinkable from output registration
  • Requires equal-denomination outputs (0.1 BTC standard)
  • Equal outputs are obvious in the transaction graph
  • Change from rounding leaks identity

WabiSabi (2020 paper → Wasabi v2, 2022)

  • Replace blind sigs with KVAC (keyed-verification anonymous credentials)
  • Credentials encode value commitments — not fixed denominations
  • Coordinator verifies balance without learning amounts
  • Variable-amount outputs
  • More flexible, harder to fingerprint

Paper: Ficsor, Kogman, Ontivero, Seres — "WabiSabi: Centrally Coordinated CoinJoins with Variable Amounts" (2020)

WabiSabi: KVAC Mechanics

KVAC = MAC-based credentials over elliptic curve groups, with three key properties:

Blind issuance

Coordinator issues a credential over a committed amount without seeing the opening. Based on a MAC scheme (Chase-Meiklejohn-Zaverucha over a group).

Unlinkable presentation

Before presenting, user re-randomizes the credential. Coordinator cannot link a presentation back to the issuance. Breaks input → output linkage.

Homomorphic amounts

Amounts are Pedersen commitments: C = v·G + r·H. Coordinator enforces that Σ inputs = Σ outputs + fee, without learning v.

Σ Cin = Σ Cout + fee·G     (verified via homomorphic sum + ZK proof)

Range proofs (Bulletproofs) ensure v ≥ 0 on each credential — preventing inflation via negative amounts.

WabiSabi: Registration Flow

1

Input phase
Alice connects via Tor (fresh circuit), registers UTXO, commits to amount, requests a credential from coordinator.

2

Credential issuance
Coordinator verifies UTXO ownership & existence. Issues MAC-based credential over the committed amount (blind).

3

Output phase
Alice connects via a different Tor circuit, re-randomizes credential, presents it, registers output. Proves balance sum. Coordinator cannot link to phase 1.

4

Signing phase
All participants sign the complete transaction. Coordinator broadcasts.

Separate Tor circuits between phases is critical — IP linkage between input and output registration would break the entire scheme.

WabiSabi: Honest Downsides

  • Central coordinator — zkSNACKs shut down their WabiSabi coordinator in June 2024. Self-hosting is possible but most users relied on the default. Smaller anonymity sets on alternative coordinators.
  • The chain analysis partnership — discovered ~2022: zkSNACKs were using chain analysis scores to block "dirty" UTXOs from joining. Directly contradicts the privacy mission. [1]
  • Not trustless in practice — the implemented client does not verify coordinator key consistency. A malicious coordinator can issue different credential parameters to each client, partitioning them and linking inputs to outputs. [2]
  • Complexity — KVAC is significantly more complex than blind signatures. Larger cryptographic attack surface, harder to audit.

JoinMarket

Decentralized liquidity market with no central coordinator

JoinMarket: Maker / Taker Model

Makers (yield generators)

  • Post offers to a public orderbook (IRC relay / Nostr)
  • Advertise: min/max join size, fee rate, counterparty count
  • Stay online passively, waiting for takers
  • Earn small fees from each join they participate in
  • Continuously remix their own UTXOs as a side effect
  • No identity required — pseudonymous by nick

Takers

  • Browse the orderbook, select N makers
  • Specify the exact CoinJoin amount they want
  • Coordinate the join via encrypted messages
  • Pay fees to each participating maker
  • Don't need to stay online long-term
  • Output equal to selected amount

No central coordinator — the orderbook is just a message board. No single party can block a taker from participating or surveil which UTXO maps to which output.

JoinMarket: Fidelity Bonds

The Sybil problem: an attacker runs 100 fake maker identities cheaply. A taker selecting 5 makers might select 5 Sybils → privacy collapses to zero.

Solution (introduced 2021): time-locked bonds

  • Makers can lock BTC in a timelocked address (OP_CLTV)
  • Bond value is computed as roughly B² × t² (bitcoins locked × time locked, squared)
  • Takers preferentially select high-fidelity-bond makers from the orderbook
  • The locked BTC earns nothing — it's a real economic sacrifice
  • To fake N identities convincingly you need N real bonds = N × capital × time locked
fidelity_weight ∝ locked_BTC² × lock_duration² × bitcoin_price

Bond scripts and amounts are on-chain — anyone can independently verify maker fidelity without trusting anyone.

JoinMarket: Tradeoffs

Strengths

  • No central coordinator → more decentralized
  • Fidelity bonds give meaningful Sybil resistance
  • Makers are incentivized to stay online → consistent liquidity

Weaknesses

  • Cost for the taker — taker pays all mining fees for the entire multi-party transaction (N+1 inputs, N+1 outputs) plus maker fees. Multiple rounds needed for real anonymity — costs multiply. High-fee environments make it prohibitive.
  • Toxic change — if your UTXO doesn't match the join amount exactly, the leftover change is directly linked to your original UTXO. Pre-mix identity leaks through the change output.
  • Technical UX — relatively bad UI. High barrier for non-technical users.

Samourai / Whirlpool

Equal-output mixing, post-mix toolset, and what happened

Whirlpool: How It Works

Equal-output CoinJoin, 5 participants per round, free remixing.

Pool denominations

  • 100,000 sats (0.001 BTC)
  • 1,000,000 sats (0.01 BTC)
  • 5,000,000 sats (0.05 BTC)
  • 50,000,000 sats (0.5 BTC)

Tx0 (premix)

Splits your UTXO into pool-denomination outputs + coordinator fee + "doxxic change" (returned immediately, not mixed). The change is clearly identifiable.

Mix cycle

  • 5 participants: 2 fresh inputs + 3 free riders (already-mixed UTXOs)
  • Free remixing: once mixed, a UTXO re-enters at no cost → growing anonymity set
  • Remixing is passive/background while wallet is connected

Post-mix tools

  • Stowaway PayJoin between Samourai users
  • StonewallX2 2-person collaborative spend
  • Ricochet extra hops to destination (defeats "blockchain distance" heuristics)

The Samourai Arrests

April 24, 2024: Keonne Rodriguez (CEO) and William Lonergan Hill (CTO) arrested.

SDNY charges

  • Conspiracy to commit money laundering (18 U.S.C. § 1956)
  • Operating an unlicensed money transmitting business (18 U.S.C. § 1960)
  • Alleged: $2B+ transactions processed; $100M+ from darknet markets

The legal theory

DOJ argued Samourai was a "money transmitter" because it routed others' funds — even though it never took custody.

This directly challenges the non-custodial model: if routing funds through a protocol you built counts as money transmission, what else does?

  • Servers and domain (samourai.io) seized; app removed from Google Play
  • Rodriguez was arrested at Barcelona airport
  • Case ongoing as of 2026

What Survives After Samourai

ASHIGARU

Community fork of Samourai Wallet, developed and maintained independently. Japanese-led team. Available on GitLab. Whirlpool coordinator separate from the seized original.

The code

Samourai Wallet and Whirlpool are open source (Apache 2.0). The tools don't disappear with the company — but the default coordinator and the user base have fragmented.

The broader effect: chilling. Other Bitcoin privacy developers paused work or shifted jurisdiction after April 2024.

The Fungibility Threat

How they detect it

  • Blockchain analytics firms: Chainalysis, CipherTrace, Elliptic
  • Detect CoinJoin patterns: equal outputs, coordinator addresses, Whirlpool structure
  • Assign "risk scores" to UTXOs — exchanges act on the score

Why it matters

  • Users penalized for receiving coins that were previously mixed by someone else
  • CoinJoin is legal — no law prohibits it. Exchanges act preemptively.
  • Creates two classes of bitcoin: "clean" and "tainted" — undermining fungibility
  • The Binance "promise" incident: behavioral compliance for using a legal tool

Tornado Cash

The Ethereum privacy case — sanctions, arrests, and legal precedent

Tornado Cash: What Happened

A non-custodial, smart-contract mixer on Ethereum. Users deposit ETH/tokens into a pool and withdraw to a fresh address using a zero-knowledge proof — breaking the on-chain link.

OFAC sanctions (Aug 2022)

  • U.S. Treasury sanctioned Tornado Cash smart contract addresses
  • First time OFAC sanctioned autonomous code rather than a person or entity
  • All U.S. persons prohibited from interacting with the contracts
  • GitHub repos removed; developer RPC endpoints blocked access

The arrests

  • Alexey Pertsev — arrested in Netherlands (Aug 2022). Convicted of money laundering by Dutch court (May 2024). Sentenced to 64 months. Released pending appeal.
  • Roman Storm — arrested in U.S. (Aug 2023). Trial Aug 2025: convicted of unlicensed money transmission; jury deadlocked on money laundering & sanctions charges. [6]
  • Roman Semenov — indicted, remains at large.

Tornado Cash: The Legal Aftermath

Fifth Circuit ruling (Nov 2024)

  • Court ruled OFAC overstepped its authority
  • Immutable smart contracts are not "property" under IEEPA — no one owns or controls them
  • "A vending machine has an owner; Tornado Cash's smart contracts do not."

Treasury delisting (Mar 2025)

  • Treasury lifted sanctions on Tornado Cash rather than appeal [7]
  • Texas court permanently enjoined OFAC from re-sanctioning (Apr 2025) [8]
  • Acknowledged "novel legal and policy issues"

Parallels with Samourai

  • Same DOJ theory: writing non-custodial privacy software = operating a money transmitter
  • Storm's mixed verdict shows juries struggle with this framing
  • Sanctions overturned, but criminal cases continue — developers still face prison for writing code
  • Combined chilling effect on all crypto privacy development, Bitcoin and Ethereum alike

Honest Conclusions

What's still missing

  • Amount privacy — all three CoinJoin implementations leave amounts visible. Confidential Transactions would fix this; not on Bitcoin mainnet.
  • Anonymity set size — privacy tools need a crowd. Current user bases are small.
  • Post-mix discipline — users routinely merge outputs and undo all gains
  • Legal clarity — if non-custodial privacy infrastructure = money transmission, the next decade looks difficult for builders
